<objective>
Fix the service worker push event handler so iOS WebKit delivers background push notifications reliably.

Purpose: iOS silently suppresses notifications when the options object includes `badge` (unsupported on iOS WebKit) or `renotify: true`. Additionally, Phase 2 D-02 specified a focus-check guard (skip OS toast when the PWA is already focused) that was never implemented. Without these fixes, iOS users do not receive push notifications when the PWA is backgrounded.

Output: Edited `frontend/public/sw.js` push handler with iOS-incompatible options removed and a focus-check guard wrapping `showNotification`.

    Replace the single line:
    ```js
    return self.registration.showNotification(title, options);
    ```

    With this exact block:
    ```js
    const windowClients = await self.clients.matchAll({ includeUncontrolled: true, type: 'window' });
    const focused = windowClients.some(c => c.visibilityState === 'visible');
    if (!focused) {
        return self.registration.showNotification(title, options);
    }
    return Promise.resolve();
    ```

    Per D-06 and Phase 2 D-02: when the PWA is open and visible, the in-app WebSocket toast (Dashboard.jsx lines 26-69) already handles the alert, so the OS toast is redundant. When backgrounded (no client in 'visible' state — typical iOS background case), `showNotification` fires.

    CRITICAL — do NOT omit the trailing `return Promise.resolve();`. The async IIFE is wrapped in `event.waitUntil()`, which requires a settled promise even on the focused (skip) branch. Without it, the service worker may be terminated mid-flight or log unhandled-promise warnings.

    Do NOT modify any other part of the push handler (title/options derivation, try/catch). Do NOT touch the `notificationclick` handler. Do NOT touch the `fetch` or `install` handlers.
