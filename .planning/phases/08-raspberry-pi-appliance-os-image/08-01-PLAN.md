<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host network → containers | All containers share the host network namespace; any container can bind any host port |
| containers → /dev/bus/usb | sdr-dsp has direct USB device access (privileged + devices mount) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Spoofing | localhost binding (api-broker on :8000) | accept | Single-tenant appliance; localhost is the host loopback only reachable from on-device processes and (via host network) other containers. No multi-user threat model. |
| T-08-02 | Information Disclosure | api-broker bound to all interfaces by default with host networking | mitigate | api-broker should bind to 0.0.0.0:8000 (PWA needs LAN access for navtex.local). Risk: any device on LAN can hit /messages. Mitigation deferred to future auth phase; documented as accepted in single-trusted-LAN appliance model. |
| T-08-03 | Elevation of Privilege | privileged: true on sdr-dsp | accept | Required for SDRPlay USB API. Standard pattern for SDR appliances; container is not exposed to remote actors. |
| T-08-04 | Tampering | docker-compose.yml on rootfs | mitigate | Image is shipped read-mostly; modifications require root + reboot. SD card tampering is the user's physical-access risk; out of scope. |
</threat_model>
