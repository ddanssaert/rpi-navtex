**image/scripts/generate-manifest.sh** (executable, mode 0755):
```bash
#!/usr/bin/env bash
# Generate rpi-imager.json from rpi-imager.json.template after a successful build.
# Required env: TAG, REPO (e.g. "yourorg/rpi-navtex"), IMG_PATH (path to navtex.img.xz)
set -euo pipefail

: "${TAG:?TAG required (e.g. v0.1.0)}"
: "${REPO:?REPO required (e.g. owner/rpi-navtex)}"
: "${IMG_PATH:?IMG_PATH required (path to navtex.img.xz)}"

HERE="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
TEMPLATE="$HERE/../rpi-imager.json.template"
OUT="${OUT:-$HERE/../../rpi-imager.json}"

# Compressed (.img.xz) — what the user downloads.
DOWNLOAD_SIZE=$(stat -c%s "$IMG_PATH")
DOWNLOAD_SHA256=$(sha256sum "$IMG_PATH" | awk '{print $1}')

# Uncompressed — Imager needs this to size the SD card and verify post-write.
EXTRACT_SIZE=$(xz --robot --list "$IMG_PATH" | awk '/^totals/ {print $5}')
EXTRACT_SHA256=$(xz -dc "$IMG_PATH" | sha256sum | awk '{print $1}')

RELEASE_DATE=$(date -u +%Y-%m-%d)

sed \
  -e "s|__TAG__|${TAG}|g" \
  -e "s|__REPO__|${REPO}|g" \
  -e "s|__EXTRACT_SIZE__|${EXTRACT_SIZE}|g" \
  -e "s|__EXTRACT_SHA256__|${EXTRACT_SHA256}|g" \
  -e "s|__DOWNLOAD_SIZE__|${DOWNLOAD_SIZE}|g" \
  -e "s|__DOWNLOAD_SHA256__|${DOWNLOAD_SHA256}|g" \
  -e "s|__RELEASE_DATE__|${RELEASE_DATE}|g" \
  "$TEMPLATE" > "$OUT"

# Verify it parses as JSON.
jq . "$OUT" > /dev/null

echo "Generated $OUT"
echo "  download size:    $DOWNLOAD_SIZE"
echo "  download sha256:  $DOWNLOAD_SHA256"
echo "  extract size:     $EXTRACT_SIZE"
echo "  extract sha256:   $EXTRACT_SHA256"
```

Create `.github/workflows/build-image.yml`. Trigger only on `v*` tag pushes (D-04). Pipeline must:

1. Checkout repo (with submodules off, full clone needed for the source tree).
2. Free disk space (RPi OS images are big; ubuntu-latest runners have limited free space).
3. Run `usimd/pi-gen-action@v1` with our custom stage.
4. Locate the produced `.img.xz` in `deploy/`, generate sha256 sidecar.
5. Run `image/scripts/generate-manifest.sh` to produce `rpi-imager.json`.
6. Create a GitHub Release for the tag and upload `navtex.img.xz`, `navtex.img.xz.sha256`, `rpi-imager.json`.

Exact contents:

```yaml
name: Build Navtex Appliance Image

on:
  push:
    tags:
      - 'v*'
  workflow_dispatch:
    inputs:
      tag:
        description: 'Manual tag (e.g. v0.0.0-dev) — used only for workflow_dispatch'
        required: false
        default: 'v0.0.0-dev'

permissions:
  contents: write   # required for Releases upload

jobs:
  build-image:
    runs-on: ubuntu-latest
    timeout-minutes: 90
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Resolve tag
        id: tag
        run: |
          if [ "${{ github.event_name }}" = "push" ]; then
            echo "tag=${GITHUB_REF#refs/tags/}" >> "$GITHUB_OUTPUT"
          else
            echo "tag=${{ inputs.tag }}" >> "$GITHUB_OUTPUT"
          fi

      - name: Free runner disk space
        run: |
          sudo rm -rf /usr/share/dotnet /usr/local/lib/android /opt/ghc \
                      /opt/hostedtoolcache /usr/local/share/boost
          docker system prune -af || true
          df -h

      - name: Stage navtex source for chroot
        # 03-stack/00-run-chroot.sh expects the navtex source at
        # image/stage-navtex/03-stack/files/navtex-src so it can `docker compose build`.
        run: |
          mkdir -p image/stage-navtex/03-stack/files/navtex-src
          rsync -a --exclude='.git' --exclude='image' --exclude='.planning' \
                --exclude='node_modules' --exclude='deploy' --exclude='work' \
                ./ image/stage-navtex/03-stack/files/navtex-src/

      - name: Build image with pi-gen
        id: pigen
        uses: usimd/pi-gen-action@v1
        with:
          image-name: navtex
          pi-gen-version: arm64
          stage-list: stage0 stage1 stage2 ./image/stage-navtex
          pi-gen-dir: /tmp/pi-gen
          increase-runner-disk-size: true
          enable-ssh: 1
          locale: en_GB.UTF-8
          keyboard-keymap: gb
          keyboard-layout: "English (UK)"
          timezone: Europe/London
          hostname: navtex
          username: navtex
          compression: xz
          compression-level: 6

      - name: Locate built image
        id: img
        run: |
          set -euo pipefail
          IMG=$(find /tmp/pi-gen/deploy -name '*.img.xz' | head -n1)
          if [ -z "$IMG" ]; then
            echo "ERROR: no .img.xz produced under /tmp/pi-gen/deploy" >&2
            ls -laR /tmp/pi-gen/deploy || true
            exit 1
          fi
          # Normalise filename to navtex.img.xz for stable URLs.
          cp "$IMG" "$GITHUB_WORKSPACE/navtex.img.xz"
          echo "path=$GITHUB_WORKSPACE/navtex.img.xz" >> "$GITHUB_OUTPUT"

      - name: Generate SHA256 checksum
        run: |
          cd "$GITHUB_WORKSPACE"
          sha256sum navtex.img.xz > navtex.img.xz.sha256
          cat navtex.img.xz.sha256

      - name: Generate rpi-imager.json
        env:
          TAG: ${{ steps.tag.outputs.tag }}
          REPO: ${{ github.repository }}
          IMG_PATH: ${{ steps.img.outputs.path }}
          OUT: ${{ github.workspace }}/rpi-imager.json
        run: |
          chmod +x image/scripts/generate-manifest.sh
          ./image/scripts/generate-manifest.sh
          cat "$OUT"

      - name: Upload artifacts (workflow run)
        uses: actions/upload-artifact@v4
        with:
          name: navtex-image-${{ steps.tag.outputs.tag }}
          path: |
            navtex.img.xz
            navtex.img.xz.sha256
            rpi-imager.json
          retention-days: 14

      - name: Create GitHub Release and upload assets
        if: github.event_name == 'push'
        uses: softprops/action-gh-release@v2
        with:
          tag_name: ${{ steps.tag.outputs.tag }}
          name: ${{ steps.tag.outputs.tag }}
          fail_on_unmatched_files: true
          files: |
            navtex.img.xz
            navtex.img.xz.sha256
            rpi-imager.json
          body: |
            Navtex Appliance Image — ${{ steps.tag.outputs.tag }}

            **How to flash:**
            1. Open Raspberry Pi Imager.
            2. Choose Device → Raspberry Pi 3.
            3. Choose OS → "Use custom" or paste this URL into "Add a custom repository":
               `https://github.com/${{ github.repository }}/releases/latest/download/rpi-imager.json`
            4. Click the gear icon → fill in WiFi, hostname, SSH key.
            5. Flash. The Pi boots zero-config; reach it at `http://navtex.local`.

            **Fallback hotspot:** If no known WiFi is reachable, the Pi opens an open AP `Navtex-AP` (192.168.99.1).

            SHA256 checksum: see `navtex.img.xz.sha256`.
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Docker apt repo | External package fetch during chroot stage |
| GitHub Actions runner → AccessPopup repo | External git clone during chroot stage |
| GitHub Releases → end users | Image artifacts distributed publicly |
| RPi Imager → JSON manifest URL | TLS-bound integrity of the manifest path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-20 | Tampering | navtex.img.xz integrity in transit | mitigate | Publish `navtex.img.xz.sha256` as a sibling Release asset; manifest contains both `image_download_sha256` (compressed) and `extract_sha256` (uncompressed). Imager verifies SHA256 automatically. |
| T-08-21 | Spoofing | rpi-imager.json URL | mitigate | URL is `https://github.com/<repo>/releases/.../rpi-imager.json` — TLS via GitHub. Pinning the URL to a tag (not `latest`) is supported via the template's TAG substitution. |
| T-08-22 | Elevation of Privilege | `permissions: contents: write` in workflow | accept | Required for Release creation. Scope is per-workflow, not org-wide. Risk: a compromised PR that modifies the workflow could push assets — mitigated by branch protection on `main` (operational control, not in this file). |
| T-08-23 | Tampering | softprops/action-gh-release@v2 supply chain | mitigate | Pinned to major version `v2`; consider pinning to a commit SHA in a follow-up hardening pass. Action runs only on tag pushes which require write access to the repo. |
| T-08-24 | Tampering | usimd/pi-gen-action@v1 supply chain | accept | Same posture as above; pinned to major version. Action only runs in CI on tag pushes. |
| T-08-25 | Information Disclosure | Image artifacts public | accept | Project is open-source; image contains no secrets (Imager-injected SSH key + WiFi PSK are written by the END USER post-flash, not baked into the image). |
| T-08-26 | DoS | 30-min build × tag-triggered only | mitigate | D-04 explicitly tag-triggers (not per-commit). `timeout-minutes: 90` caps runaway builds. |
</threat_model>
